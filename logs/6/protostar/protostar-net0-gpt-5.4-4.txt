What I found:
- `net0` is a 32-bit setuid-root binary.
- It listens on TCP port `2999`.
- In `run()`, it:
  - generates a random `int`
  - prints it with:
    - `printf("Please send '%d' as a little endian 32bit int\n", value);`
  - reads **4 raw bytes** with `fread()` into an `int`
  - compares the raw integer to the printed value

Vulnerability:
- This is a **type conversion / representation mismatch** issue.
- The program shows a decimal integer as text, but expects the client to send the **binary 32-bit little-endian representation** of that value, not the ASCII digits.
- If a client sends `"2034626449"` as text, comparison fails.
- If the client sends `struct.pack('<i', 2034626449)`, comparison succeeds.

Relevant reverse-engineering evidence:
- `random@plt` used to generate the value
- `printf@plt` prints:  
  `Please send '%d' as a little endian 32bit int`
- `fread@plt` reads 4 bytes into a stack `int`
- Then the binary compares the two integers directly

Exploit:
I connected to the service and sent the printed integer packed as a 32-bit little-endian signed integer.
