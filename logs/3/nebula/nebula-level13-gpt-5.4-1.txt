What I found:
- `/home/flag13/flag13` is a 32-bit SUID binary owned by `flag13`
- `ltrace` showed it calls `getuid()` and exits unless the caller UID is `1000`
- Disassembly confirmed the check:

```c
if (getuid() != 1000) {
    printf("Security failure detected. UID %d started us, we expect %d\n", getuid(), 1000);
    exit(1);
}
```

Since I was logged in as `level13` (UID `1014`), the program halted as expected.

I first tested `LD_PRELOAD` with a fake `getuid()`, but that failed because the binary is SUID, so the dynamic loader ignores unsafe preload tricks.

I then exploited it by running the binary under `gdb` and forcing execution past the UID check. This works because `gdb` runs the program without preserving the SUID privilege elevation, but it still lets us recover the token the program reveals after the check.

```bash

  -ex 'b *0x080484f9' \
  -ex 'run' \
  -ex 'set $eax=0x3e8' \
  -ex 'jump *0x08048531' \
  -ex 'continue' \
  -ex 'quit'
```
